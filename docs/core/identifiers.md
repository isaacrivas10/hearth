# Spec: `EntityId`

> **Source module**: `hearth/identifiers.py`
> **Parent ADRs**: [ADR-0005 (primitives)](../architecture/0005-primitives.md) (with 2026-05-02 amendment), [ADR-0007 (transactional contract)](../architecture/0007-transactional-contract.md), [ADR-0008 (built-in plugins)](../architecture/0008-built-in-plugins.md).

## Status

Stub. `EntityId` is one of two kernel-shipped value types (the other is
`Reference`).

## Brief

The kernel-assigned identity for entities. ULID-backed by default for
monotonic ordering and outbox-friendly sorting (see ADR-0007 §"Outbox
order is preserved per aggregate").

Assigned at first persist, not at construction. Plugin authors do not
generate or assign EntityIds themselves.

## Sketch

```python
from hearth import EntityId

# Generated by the kernel on uow.save:
customer.id          # EntityId
str(customer.id)     # "01HV3K7QY9C..."  (ULID Crockford-base32 form)
```

## Open questions

- **Typed aliases per entity.** `CustomerId = NewType('CustomerId', EntityId)`
  for stronger typing on `References` and action commands. Likely yes;
  the slice will tell us how it feels.
- **Backing format.** ULID (current lean) vs. UUIDv7 vs. autoincrement.
  ULID/UUIDv7 win for outbox ordering; autoincrement is denser but
  loses ordering across distributed writers.
- **Wire form for serialization.** String, bytes, or native UUID column.
- **Pre-save IDs.** Whether plugin authors can opt into client-side ID
  generation (rare; needed for some idempotency patterns).
